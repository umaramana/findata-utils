# S3.3 — WhatsApp Nudge (scoping note, 2026-06-25 — not yet built)

**Status: stub.** This card is not a build spec — S3.3 hasn't been scoped yet (roadmap slot: Thu, per `project_insight_core.md` sprint plan). This file exists to lock in one requirement now, before S3.3 design starts, so it isn't designed around the wrong trigger.

**Why this note exists**
Insight is moving to a continuous-assessment model: Arun logs 1-2 tests in a session, then wants to nudge the client on WhatsApp immediately — not wait until a client's full 10-section assessment is complete across however many sessions that takes. Today, `submitFullAssessment` (`Code.gs`) already supports partial-day saves correctly (only filled metrics write, returns `{saved, removed}` counts per F02-S02). The nudge itself doesn't exist yet — zero messaging/WhatsApp code in the repo as of this note — so there's nothing currently *wrong* to fix, but there's a wrong assumption to avoid baking in later.

**Locked requirement for S3.3's eventual design**
- The nudge trigger must be scoped to **"the metrics saved in one `submitFullAssessment` call"**, not to "the full assessment is complete."
- Natural hook: `submitFullAssessment`'s existing return value (`saved`/`removed` counts, and the metric_ids that changed) is exactly the right input for "what should this nudge talk about" — reuse it rather than re-querying `readings` for "what's new" from scratch.
- Do not design S3.3 around a full-assessment-completion gate (e.g. "all 9 active components have at least one reading") — that contradicts how Arun actually works and was the original framing this note corrects.

**Explicitly not decided here** (leave for S3.3's actual scoping session): nudge content/template, PNG generation, WhatsApp delivery mechanism (manual share vs API), how often Arun expects to send one, trigger UI (auto-prompt after every save vs explicit "Send Nudge" button).
